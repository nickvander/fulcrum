### High-Level Goal

To create a single, centralized, and well-organized documentation hub that serves as the source of truth for all developers. We will consolidate scattered information, adopt a clearer structure, enhance the content to be more comprehensive, and ensure all documentation is grounded in the actual implementation of the `backend` and `frontend` code.

---

### **Phase 1: Triage and Consolidation**

The first step is to address the fragmentation and redundancy of the existing documentation.

1.  **Centralize All Developer Docs:** The information in `README.md`, `CONTRIBUTING.md`, and `docs/README.md` is essential but scattered. We will merge the core content of these files into the main Sphinx documentation located in `docs/source/`.
    *   **`CONTRIBUTING.md`**: Its content will be moved to a new `contributing.md` file within the Sphinx docs. The original file will be replaced with a simple link to the new location.
    *   **`docs/README.md`**: The instructions on how to build the documentation are a form of contribution. This content will be merged into the new `contributing.md` section.
    *   **Root `README.md`**: This file will be simplified to serve as a high-level project entry point, directing users to the comprehensive Sphinx documentation for all technical details.

2.  **Eliminate Redundancy:** The files `docs/source/README.md` and `docs/source/introduction.md` serve very similar purposes.
    *   We will merge them into a single, welcoming `introduction.md` that will serve as the landing page for the documentation site.
    *   The manual table of contents in these files will be removed, as the navigation sidebar generated by Sphinx is the source of truth.

### **Phase 2: Restructure and Reorganize**

With the content consolidated, we will reorganize it for clarity and ease of use. The current flat structure in `docs/source/` can be improved by grouping related topics.

1.  **Adopt a Thematic Structure:** We will create subdirectories within `docs/source/` to group documentation by its purpose. I propose the following structure:

    *   `docs/source/getting-started/`: For essential setup guides.
    *   `docs/source/guides/`: For practical, step-by-step "how-to" documents.
    *   `docs/source/explanation/`: For deep dives into architecture and concepts.
    *   `docs/source/reference/`: For technical references like API specs or configuration variables (a future enhancement).

2.  **Relocate Existing Documents:** We will move the existing files into this new structure:
    *   `introduction.md` -> `docs/source/introduction.md` (will be the new root)
    *   `backend-setup.md` -> `docs/source/getting-started/backend-setup.md`
    *   `frontend-setup.md` -> `docs/source/getting-started/frontend-setup.md`
    *   `database-migrations.md` -> `docs/source/guides/database-migrations.md`
    *   `testing-and-ci.md` -> `docs/source/guides/testing-and-ci.md`
    *   `architecture.md` -> `docs/source/explanation/architecture.md`
    *   The new `contributing.md` -> `docs/source/guides/contributing.md`

3.  **Update Navigation:** The master navigation file, `docs/source/index.rst`, will be updated to reflect this new, organized, and nested structure.

### **Phase 3: Code-Driven Content Enhancement**

After restructuring, we will focus on improving the content and filling in the gaps, using the codebase as the primary source of truth.

1.  **Fix Documentation Title:** As requested, I will modify `docs/source/conf.py`. The title is generated from the `project` and `release` variables. I will update the configuration to ensure the main title is simply "Fulcrum" and not "Fulcrum 1.0".

2.  **Audit and Enhance Backend Documentation:** I will systematically review the `backend/` directory and update the documentation to reflect the code.
    *   **`explanation/architecture.md`**: I will review `backend/src/` to verify that the descriptions of the Repository Pattern, Service Layer, and Dependency Injection are accurate. I'll check `backend/src/api/v1/api.py` to confirm how routing is structured. I'll also look at `backend/src/main.py` and `backend/src/celery_worker.py` to ensure the descriptions of application startup and async tasks are correct.
    *   **`getting-started/backend-setup.md`**: I will review `docker-compose.yml` and `backend/Dockerfile` to ensure the setup instructions are accurate. I will also cross-reference `backend/.env.example` to create a new **`reference/configuration.md`** page explaining all environment variables.
    *   **`guides/database-migrations.md`**: I will examine `backend/alembic/` and `backend/migrate.sh` to confirm the migration workflow is described correctly.
    *   **`guides/testing-and-ci.md`**: I will review `backend/tests/`, `pytest.ini`, and the relevant `.github/workflows/` files to ensure the testing documentation is up-to-date.

3.  **Audit and Enhance Frontend Documentation:** I will systematically review the `frontend/` directory.
    *   **`explanation/architecture.md`**: I will add a new section for the frontend architecture. I'll review `frontend/src/app/` to document the module structure (`core`, `shared`, feature modules), state management approach (RxJS), and key components.
    *   **`getting-started/frontend-setup.md`**: I will review `frontend/package.json` and `frontend/angular.json` to verify the setup, build, and test commands. I will also check `proxy.conf.json` and explain how the frontend dev server communicates with the backend API.
    *   **PWA Capabilities**: I will review `ngsw-config.json` and `manifest.webmanifest` to add a section explaining the Progressive Web App features and configuration.

4.  **Review and Refine Content:** Throughout the process, I will:
    *   Ensure clarity, conciseness, and a consistent tone.
    *   Update any outdated information.
    *   Fix all internal links that will break after the file reorganization.
